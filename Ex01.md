## Đáp án lựa chọn: **B**

Phương án **B** là prompt tối ưu nhất.

---

## 1. Vì sao phương án B tối ưu nhất?

Phương án B:

> “Hãy đóng vai trò là một Java Senior Developer. Nhiệm vụ của bạn là tái cấu trúc logic rẽ nhánh lồng nhau phức tạp của class DiscountService trên thành các câu lệnh điều kiện bảo vệ guard clauses / return sớm. Ràng buộc: Giữ nguyên logic nghiệp vụ tính chiết khấu ban đầu, không thay đổi kiểu dữ liệu đầu vào/đầu ra, sử dụng Java 11. Trình bày kết quả dưới dạng khối mã nguồn Java hoàn chỉnh kèm giải thích ngắn bằng tiếng Việt.”

Đây là prompt tốt nhất vì có đầy đủ **5 thành phần của một Prompt hiệu quả**:

### 1. Vai trò

Prompt có chỉ định rõ vai trò:

> “Hãy đóng vai trò là một Java Senior Developer.”

Điều này giúp AI hiểu rằng cần xử lý bài toán với tư duy của một lập trình viên Java có kinh nghiệm, ưu tiên clean code, dễ bảo trì và hạn chế thay đổi logic nghiệp vụ.

---

### 2. Mục tiêu

Prompt nêu rõ nhiệm vụ chính:

> “Tái cấu trúc logic rẽ nhánh lồng nhau phức tạp của class DiscountService thành các câu lệnh điều kiện bảo vệ guard clauses / return sớm.”

Mục tiêu này rất cụ thể. AI không chỉ được yêu cầu “sửa code” chung chung mà được định hướng rõ phải dùng **guard clauses** để giảm nested if-else.

---

### 3. Ngữ cảnh

Prompt có ngữ cảnh rõ ràng:

> “Logic rẽ nhánh lồng nhau phức tạp của class DiscountService.”

Ngữ cảnh cho AI biết đoạn code hiện tại đang gặp vấn đề về nhiều `if-else` lồng nhau, khó đọc và khó bảo trì. Từ đó AI có thể đưa ra cách refactor phù hợp.

---

### 4. Ràng buộc

Prompt B đưa ra các ràng buộc rất quan trọng:

> “Giữ nguyên logic nghiệp vụ tính chiết khấu ban đầu.”

> “Không thay đổi kiểu dữ liệu đầu vào/đầu ra.”

> “Sử dụng Java 11.”

Đây là phần rất quan trọng trong bài toán refactor. Khi tái cấu trúc code, mục tiêu không phải là thay đổi nghiệp vụ mà chỉ làm code dễ đọc hơn. Các ràng buộc này giúp hạn chế việc AI tự ý thay đổi logic tính chiết khấu hoặc dùng cú pháp không phù hợp với phiên bản Java yêu cầu.

---

### 5. Định dạng đầu ra

Prompt B yêu cầu rõ kết quả cần trả về:

> “Trình bày kết quả dưới dạng khối mã nguồn Java hoàn chỉnh kèm giải thích ngắn bằng tiếng Việt.”

Điều này giúp đầu ra có cấu trúc rõ ràng, dễ đọc, dễ nộp bài và dễ kiểm tra.

---

## 2. Ví dụ kết quả AI có thể tạo ra từ prompt B

```java
public class DiscountService {

    public double calculateDiscount(String type, double amount, boolean isHoliday) {
        if ("VIP".equals(type)) {
            if (amount <= 1000) {
                return amount * 0.15;
            }

            if (isHoliday) {
                return amount * 0.25;
            }

            return amount * 0.20;
        }

        if (amount <= 500) {
            return 0;
        }

        if (isHoliday) {
            return amount * 0.10;
        }

        return amount * 0.05;
    }
}
```

Code sau khi refactor đã giảm bớt độ lồng nhau của `if-else`, dễ đọc hơn nhưng vẫn giữ nguyên logic cũ:

- Khách hàng `VIP`, amount > 1000, ngày lễ: giảm 25%.
- Khách hàng `VIP`, amount > 1000, không phải ngày lễ: giảm 20%.
- Khách hàng `VIP`, amount <= 1000: giảm 15%.
- Khách hàng không phải `VIP`, amount > 500, ngày lễ: giảm 10%.
- Khách hàng không phải `VIP`, amount > 500, không phải ngày lễ: giảm 5%.
- Các trường hợp còn lại: giảm 0.

Ngoài ra, việc dùng `"VIP".equals(type)` cũng an toàn hơn `type.equals("VIP")` vì tránh lỗi `NullPointerException` nếu `type` bị `null`.

## 3. Lý do loại trừ phương án A

Phương án A:

> “Tái cấu trúc code Java trên để nó đẹp hơn.”

Phương án A không tối ưu vì quá chung chung.

### Nhược điểm

Thứ nhất, phương án A **không có vai trò cụ thể**. AI không biết cần xử lý với tư duy của Junior, Senior Developer hay Code Reviewer.

Thứ hai, mục tiêu “đẹp hơn” rất mơ hồ. AI có thể hiểu “đẹp hơn” theo nhiều cách khác nhau như đổi tên biến, tách hàm, dùng switch-case, dùng enum, hoặc thay đổi toàn bộ cấu trúc code.

Thứ ba, phương án A **không yêu cầu dùng guard clauses**, trong khi đề bài yêu cầu phải tái cấu trúc sang dạng điều kiện bảo vệ.

Thứ tư, phương án A **không có ràng buộc bảo toàn logic nghiệp vụ**. Điều này có thể khiến AI vô tình làm thay đổi kết quả tính chiết khấu.

Thứ năm, phương án A **không quy định định dạng đầu ra**, nên AI có thể trả lời thiếu code hoàn chỉnh, thiếu giải thích hoặc trình bày không đúng yêu cầu.

Vì vậy, phương án A dễ tạo ra kết quả không đúng trọng tâm và khó kiểm soát chất lượng đầu ra.

## 4. Lý do loại trừ phương án C

Phương án C:

> “Hãy sửa code Java này, bỏ bớt if-else lồng nhau đi và sử dụng cấu trúc Java Stream API để viết ngắn gọn nhất có thể.”

Phương án C có vẻ cụ thể hơn A nhưng vẫn không phù hợp với yêu cầu đề bài.

### Nhược điểm

Thứ nhất, đề bài yêu cầu sử dụng **guard clauses / return sớm**, nhưng phương án C lại yêu cầu sử dụng **Java Stream API**. Đây là hướng đi sai so với yêu cầu chính.

Thứ hai, bài toán này là bài toán rẽ nhánh điều kiện đơn giản, không phải xử lý collection hoặc luồng dữ liệu. Vì vậy, dùng Stream API là không cần thiết và có thể làm code khó hiểu hơn.

Thứ ba, yêu cầu “viết ngắn gọn nhất có thể” có thể làm AI ưu tiên sự ngắn gọn thay vì sự rõ ràng. Điều này đi ngược lại tinh thần Clean Code, vì code tốt không chỉ ngắn mà còn phải dễ đọc, dễ hiểu và dễ bảo trì.

Thứ tư, phương án C không nhấn mạnh việc **giữ nguyên logic nghiệp vụ cũ**. Khi cố gắng viết ngắn bằng Stream API, AI có thể làm thay đổi điều kiện tính chiết khấu hoặc bỏ sót một số trường hợp.

Thứ năm, Stream API không phù hợp tự nhiên với logic dạng:

```java
if condition A
    return result A
else if condition B
    return result B
```

Nếu ép dùng Stream API, code có thể trở nên phức tạp, khó debug và không còn đúng mục tiêu refactor bằng guard clauses.
